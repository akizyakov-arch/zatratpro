# CHANGELOG 0.4 -> 0.5.4

## Пользовательские изменения
- Меню и callback-навигация ускорены: кнопки в `Помощь`, `nav:main` и смежных сценариях отвечают заметно быстрее.
- Для Telegram-трафика добавлена поддержка SOCKS5 proxy, что сняло сетевые задержки VPS -> Telegram.
- `Мои документы` переведены в drill-down меню:
  - список документов кнопками
  - карточка документа
  - отдельный экран `Состав документа`
- `Мои документы` доступны и сотруднику, и manager.
- В карточке документа появился просмотр исходного файла через кнопку `Открыть документ`.
- Загрузка документов снова стабильно работает для фото и для image-файлов, присланных как вложение.

## Документы / OCR / LLM
- Добавлены stage-тайминги document pipeline:
  - download
  - OCR
  - DeepSeek extract
  - preview/send
- Оптимизирован extraction flow для DeepSeek:
  - `raw_text` больше не генерируется моделью, а подставляется локально
  - сокращен extraction prompt без изменения JSON-контракта
  - добавлены внутренние stage-тайминги DeepSeek вызова
- Исправлен document flow после storage-рефакторинга:
  - восстановлен общий helper `_process_uploaded_image(...)`
  - починена проверка `pending_documents`
  - восстановлен OCR-flow после ранних return и `NameError`
- Image-файлы, загруженные как `document`, теперь тоже уходят в OCR-flow.

## Хранение документов
- Введено persistent storage V1 для документов вне `tmp`.
- В `docker-compose.yml` добавлен mount постоянного storage:
  - `./storage:/app/storage`
- На хосте документы теперь лежат в структуре вида:
  - `storage/documents/<company_id>/<document_id>/source.jpg`
- Для фото постоянным `source` теперь сохраняется не сырой upload, а нормализованный master (`source.jpg`).
- Временный `tmp` остался промежуточной зоной только для обработки.
- Исправлен lifecycle временных файлов, чтобы нормализованный `ocr_path` не удалялся до переноса в storage.

## Метаданные source-файлов
- В `document_files` добавлены и реально сохраняются метаданные нормализации:
  - `original_file_size`
  - `stored_file_size`
  - `was_normalized`
  - `original_kind`
- Те же поля протянуты через `pending_documents`, чтобы не терять их между OCR и выбором проекта.
- Для новых документов подтверждено сохранение:
  - `original_kind = photo`
  - корректных размеров исходного и сохраненного файла
  - `was_normalized = true`

## Отчеты / Дубли документов
- Починено открытие карточки дубля в отчетах.
- Исправлен HTML-рендер duplicate report, включая корректное форматирование карточки.
- Ветки `По проектам` и `По сотрудникам` получили отдельный экран `Состав документа` вместо полного вывода всего документа в одном сообщении.
- В карточке точного дубля добавлен просмотр исходников:
  - `Открыть исходный`
  - `Открыть дубликат`
- Карточка точного дубля упрощена:
  - убраны дублирующие summary-блоки
  - оставлены две основные карточки: дубликат и исходный документ
- Из меню точных дублей убрано действие `Оставить как отдельный` вместе с обработчиком.
- Актуальный порядок действий в карточке дубля:
  1. `Открыть исходный`
  2. `Открыть дубликат`
  3. `Удалить дубликат`
  4. `Удалить исходный`
  5. `Назад к дублям`

## Архитектурные и диагностические изменения
- Добавлено подробное perf-логирование для slow updates.
- Добавлены диагностические логи для inbound non-text updates, photo/document routing и access context.
- Эти логи использовались для точечной локализации регрессий после storage-патча и возвращения рабочего document flow.

## Итог релизной линии 0.4 -> 0.5.4
- Ускорена работа меню и callback-сценариев.
- Сетевые проблемы Telegram на VPS компенсированы через SOCKS5 proxy.
- Document flow восстановлен и стабилизирован после storage-рефакторинга.
- Внедрено постоянное хранение документов на хосте.
- Для фото принят единый подход хранения: нормализованный `source.jpg` как основной master.
- Duplicate-report доведен до рабочего менеджерского сценария с просмотром обеих версий документа.
