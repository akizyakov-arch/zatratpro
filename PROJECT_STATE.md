# PROJECT_STATE

## Project
ZATRATPRO

MVP Telegram-бот для распознавания чеков, актов и накладных с привязкой к проектам и сохранением структурированных затрат в Postgres.

## Current Deployment
Рабочая версия развернута на VPS в Docker.

Инфраструктура:
- VPS
- Ubuntu
- Docker Compose
- каталог проекта на сервере: `/root/zatratpro`

Базовый deploy flow:
1. `git push origin main`
2. `git pull origin main`
3. `docker compose up -d --build`

## Implemented Now
Что реально уже внедрено и работает:
- Telegram-бот принимает фото документа
- бот скачивает фото из Telegram
- фото отправляется в OCR.Space
- OCR текст нормализуется через DeepSeek
- пользователю показывается краткая выжимка документа
- после preview бот показывает проекты кнопками
- после выбора проекта собирается JSON документа
- документ и позиции сохраняются в Postgres
- OCR.Space hardening добавлен: увеличен timeout и есть retry на `ReadTimeout`
- есть компании, участники компаний и invite-коды
- владелец бота создает компанию
- владелец бота не становится руководителем компании автоматически
- после создания компании бот выдает invite-код для первого руководителя
- руководитель компании может:
  - создавать проекты
  - архивировать проекты
  - деархивировать проекты
  - приглашать сотрудников
  - исключать сотрудников
  - смотреть участников компании
- сотрудник может работать только после входа в компанию по invite-коду

## Current Roles
Сейчас в продуктовой логике используются только такие роли:
- `owner` — владелец бота
- `manager` — руководитель компании
- `employee` — сотрудник компании

Ключевой принцип:
- `owner` не равен `manager`
- `owner` создает компанию как системную сущность
- `manager` управляет компанией внутри продукта

## Current User Flow
### Bootstrap компании
1. Владелец бота создает компанию.
2. Бот выдает invite-код для первого `manager`.
3. Будущий руководитель отправляет `/join КОД`.
4. Только после этого он получает доступ к проектам и управлению компанией.

### Документный flow
1. Пользователь с доступом к компании отправляет фото документа.
2. Бот делает OCR через OCR.Space.
3. Бот нормализует OCR-текст через DeepSeek.
4. Бот показывает краткую выжимку документа.
5. Бот показывает список проектов текущей компании.
6. Пользователь выбирает проект.
7. Бот собирает JSON документа.
8. Бот сохраняет `documents` и `document_items`.

## Current Product Constraints
Это важно зафиксировать как текущую бизнес-модель, даже если код еще не везде до нее доведен:
- один `manager` = одна компания
- одна компания = один `manager`
- одна компания = набор сотрудников
- на текущем этапе сотрудников у одного `manager` должно быть не более `10`
- проекты принадлежат компании, а не конкретному человеку
- сотрудники должны видеть только проекты своей компании
- руководитель должен видеть только свою компанию и ее проекты

## Known Problems
### Menu Structure Is Wrong
Текущая структура меню по ролям признана неверной и требует переделки.

Что не нравится:
- меню `owner`/`manager`/`employee` пока собрано эволюционно, а не как чистая продуктовая модель
- часть сценариев все еще выглядит как технический bootstrap, а не как законченный UX
- управление компанией перегружено ручным вводом `ID`
- role-based UX нужно спроектировать заново, а не только дополнять кнопками

Нужно переделать:
- отдельное чистое меню для `owner`
- отдельное рабочее меню для `manager`
- отдельное простое меню для `employee`
- меньше fallback-команд, больше сценариев кнопками
- убрать ручной ввод `ID` там, где можно сделать кнопочные действия

### Database / Domain Model Is Still Wrong
Текущая структура базы и доменной модели тоже считается неверной относительно текущего продукта.

Что именно неверно:
- модель пока слишком общая и multi-tenant в широком смысле, а текущий продуктовый режим уже уже: `1 manager = 1 company`
- логика видимости проектов и доступа должна быть жестко подчинена компании и текущему `manager`
- нужно явно закрепить, что проекты не "общие", а принадлежат конкретной компании
- нужно явно закрепить лимит сотрудников на компанию: до `10`
- нужно убедиться, что руководитель и сотрудники не видят проекты вне своей компании

Коротко:
- меню по ролям нужно переработать
- базовую доменную модель нужно упростить и довести до продукта `1 manager = 1 company`
